# Phase 6 Part 8 — Deployment Guide: Firebase (Cloud Messaging)

For obtaining dev service-account credentials, see root `docs/phase-5.md` "Provider setup guide" — this covers production-specific concerns.

## Service account credentials

`FIREBASE_PROJECT_ID`, `FIREBASE_CLIENT_EMAIL`, `FIREBASE_PRIVATE_KEY` (in `FcmPushNotificationService`) come from a Firebase service account JSON key. For production: create a **dedicated** service account scoped to Cloud Messaging only (Firebase Console → Project Settings → Service Accounts → Generate new private key), separate from any service account used for local development, so a leaked dev key can't send push notifications to real production users. Store the private key exactly as `env.ts` expects — escaped `\n` sequences in a single-line env var value, which `env.ts` already un-escapes (`.replace(/\\n/g, "\n")`).

## One Firebase project, both platforms

A single Firebase project covers both Android and iOS push delivery through Cloud Messaging — there's no separate "production Firebase for Android" vs "production Firebase for iOS" split needed on the backend side. What does differ per platform:

- **Android**: the Capacitor-built app (Part 7) needs `google-services.json` (downloaded from Firebase Console → Project Settings → your Android app, matching the `com.rentit.app` app ID in `capacitor.config.ts`) placed at `frontend/android/app/google-services.json` once the native Android project exists.
- **iOS**: push delivery via FCM to iOS requires an APNs (Apple Push Notification service) authentication key uploaded *into* the Firebase project (Firebase Console → Project Settings → Cloud Messaging → Apple app configuration) — this is an Apple Developer Program artifact (a `.p8` key from developer.apple.com), not something generated by Firebase or this codebase. Without it, FCM cannot deliver to iOS devices at all even though the backend code is identical for both platforms.

## Token lifecycle

Device push tokens are already handled correctly by the existing code (Phase 5) — `UserDeviceRepository` stores them, and a token that FCM reports as invalid/unregistered (e.g. app uninstalled) should be pruned rather than retried indefinitely. Confirm `FcmPushNotificationService`'s error handling actually removes a device row on FCM's `UNREGISTERED`/`NOT_FOUND` error codes in production traffic — this matters more at real scale than in dev testing, where token churn is minimal.

## Rate limits / quotas

FCM's free tier is generous (no hard cap on message volume for standard messages) — not a likely production blocker the way Cloudinary's storage/bandwidth limits are. No paid tier is required specifically for Cloud Messaging.
