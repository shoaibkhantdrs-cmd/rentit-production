# RentIt — Master Backlog (QA + Security + Performance + Architecture, Merged)

Consolidates `docs/qa-full-bug-report.md`, `docs/security-audit.md`, `docs/performance-audit.md`, and the architecture-relevant findings from `docs/rc1-bug-report.md` (no standalone "architecture audit" exists in this project — see note at bottom) into one severity-sorted backlog. Read-only synthesis; no code was written or changed to produce this document.

## Methodology

- **Resolved items excluded.** 22 of the 23 findings in `qa-full-bug-report.md` were already fixed and independently verified in `docs/qa-fixes-verification.md` (0 TS errors, 0 lint warnings, all 22 re-checked against source). Only the one remaining open item from that pass is carried into this backlog. `rc1-bug-report.md`'s own bug fixes (PATCH/PUT mismatch, error-swallowing, etc.) and its dead-code/duplicate-CSS cleanup are likewise excluded as resolved — only what that report explicitly left untouched is carried forward.
- **Duplicates merged.** The admin-payments/refunds frontend gap was independently flagged by both the QA pass (`qa-full-bug-report.md` #22) and the RC1 pass ("known gaps" section) — merged into one row. The security audit's Finding #12 (dev-login backdoor) shares its exact root cause with Finding #1 (JWT secret `NODE_ENV` guard) — folded into Finding #1's row as a note rather than counted twice. The performance audit's "Top 10" list and its detailed per-area sections describe the same issues at two levels of depth — each is counted once here, at its Top-10 identity, with detail-section items that aren't part of the Top 10 added separately.
- **False positives removed.** `Badge.tsx` was flagged as dead code by one internal audit pass but is live-imported by `ProfilePage.tsx:24` — excluded. The performance audit's own "confirmed correct / non-issue" sections (JWT alg-confusion, CSRF, XSS, SSRF, SQLi, tree-shaking, bundle-graph inclusion, batching patterns already done correctly, etc. — several dozen items across both the security and performance audits) are confirmations of *absence* of a problem, not findings, and are excluded from this backlog entirely. `Layout.tsx`'s `AnimatePresence mode="wait"` route-transition delay and the `pool: max 10` connection-pool note are both explicitly described in their source audit as intentional tradeoffs/contextual caveats rather than defects — excluded as non-actionable.
- **Blocking-release judgment.** Marked **Yes** for: the one Critical finding; High-severity security findings with concrete compromise potential (auth bypass, moderation bypass); and Medium-severity security findings that expose money movement or sensitive PII even though their *likelihood* is gated behind a misconfiguration, because that exact misconfiguration is confirmed present in this project's current `.env` (blank SMTP/Twilio/webhook secrets). Everything performance- and architecture-flagged is marked **No** — these affect scale and maintainability, not correctness or safety, and none currently block a functioning release.
- **Effort scale:** Trivial (<1 hr) · Small (a few hours) · Medium (1-3 days) · Large (>3 days / multi-day project).

| 1 | Auth / JWT | `backend/src/config/env.ts:23-28,56` | JWT signing secret falls back to a hardcoded, source-visible value (`dev-only-insecure-secret-change-me`) unless `NODE_ENV` is the exact string `"production"`. ## Note on "Architecture" scope

No report in this project was ever titled or scoped as a dedicated architecture audit. Per your direction, this backlog uses the closest available source — `docs/rc1-bug-report.md`'s dead-code, duplicate-CSS, and duplicate-component findings — as the Architecture input. If a genuinely broader architecture review (module boundaries, dependency direction, layering violations, coupling) is wanted, that would need to be run as its own pass; nothing in the existing audit set covers that ground.
