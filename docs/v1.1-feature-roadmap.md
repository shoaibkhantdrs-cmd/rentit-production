## 1. India Location Hierarchy

### Current state

`property_locations` (migration `1700000000017`) stores `city`, `locality`, `state`, `country`, `postal_code` as free-text columns, populated from whatever Google Maps Geocoding returns. `SearchPage.tsx`'s city/locality filters are plain text inputs matched via `ILIKE` (backed by the trigram indexes added in RC1). There is no canonical list of Indian states/cities/localities anywhere in the schema — two listings in "Bengaluru" and "Bangalore" are different strings to every query in this system today.

### Proposed data model (additive)

New reference tables, populated once from a static dataset (e.g. India Post's PIN code dataset or a maintained states/districts/cities dataset — needs a real data source decision before building, not fabricated data):

```sql
CREATE TABLE in_states (
  id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name  TEXT NOT NULL UNIQUE,       -- "Karnataka"
  code  TEXT NOT NULL UNIQUE        -- "KA"
);

CREATE TABLE in_cities (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  state_id  UUID NOT NULL REFERENCES in_states(id),
  name      TEXT NOT NULL,          -- "Bengaluru"
  UNIQUE (state_id, name)
);

-- Localities are numerous and messier than cities (thousands per city,
-- inconsistent official naming) -- start without a dedicated table and
-- instead add a nullable FK to a locality once the free-text city FK
-- migration has proven out, rather than modeling both at once.
ALTER TABLE property_locations
  ADD COLUMN state_id UUID NULL REFERENCES in_states(id),
  ADD COLUMN city_id  UUID NULL REFERENCES in_cities(id);
```

Both new FK columns are **nullable and additive** — existing rows keep their free-text `city`/`state` values untouched, and every existing query that reads those text columns keeps working unmodified. This is the same additive-migration discipline RC1's index migrations used (`ADD COLUMN`/`CREATE INDEX`, no `ALTER ... NOT NULL`, no data rewrite of existing rows).

### Backend

- New read-only endpoints: `GET /locations/states`, `GET /locations/states/:stateId/cities` — small, cacheable, no auth required (same pattern as `GET /properties/categories`).
- `CreateProperty`/`UpdateProperty` use cases accept optional `stateId`/`cityId` alongside the existing free-text address fields; when provided, back-fill `city`/`state` text columns from the resolved names so every existing consumer (search, display) keeps working unchanged whether or not the hierarchy was used for a given listing.
- `SearchProperties` gains optional `stateId`/`cityId` filters as **additional**, backward-compatible query params — existing `city=` free-text search keeps working exactly as today for any client that hasn't adopted the new dropdown filters yet.

### Frontend

- `AddPropertyPage`'s location step: replace the free-text city input with a two-level dependent dropdown (state → city), falling back to the existing free-text input for locality (and for city, if a typed city isn't in the reference list yet — never block a listing from being created just because the dataset is incomplete).
- `SearchPage`'s Location filter section: add state/city dropdowns above the existing city/locality text inputs; keep the text inputs as-is underneath for locality search, which stays free-text.

### Why this is additive, not a redesign

No existing column is renamed, dropped, or made non-nullable. No existing API response shape changes. A client that never sends `stateId`/`cityId` sees identical behavior to today.

## 2. New Property Types

### Current state

`PropertyType` (`backend/src/domain/entities/Property.ts`) is a fixed TypeScript union — `apartment | house | villa | studio | pg | room | commercial | other` — enforced by a Postgres `CHECK` constraint on the `properties.property_type` column (needs confirming the exact constraint at build time, but every other enum-like column in this schema uses `CHECK (... IN (...))`, not a Postgres native `ENUM` type, per the existing `payment_orders.purpose`/`payments.status` pattern above). Adding a new type today means a migration to widen the `CHECK` constraint plus a source change to the union type and the `PROPERTY_TYPES` array in `SearchPage.tsx` — small, but requires a deploy for every new type.

### Proposed approach

Two real options, not fabricated as one:

**Option A — widen the existing enum (small, fast, matches current architecture).** Add specific new values as requested (e.g. `farmhouse`, `co_living`, `warehouse`, `office_space` — the actual list needs the business decision on what to add, not invented here) via a migration that drops and recreates the `CHECK` constraint with the expanded list, plus the corresponding `PropertyType` union and `PROPERTY_TYPES` UI array update. Zero data migration needed (existing rows are unaffected; only new inserts can use new values). This keeps the current architecture and is the recommended default unless the real requirement is "owners/admins can add types without a deploy."

**Option B — move property types to a reference table (bigger, only if that's the real requirement).** A `property_types` table (`id`, `slug`, `label`, `is_active`) with `properties.property_type` becoming a foreign key instead of a `CHECK`-constrained string. This is the only way to add a type without a code deploy, but it's a materially bigger change (touches every `PropertyType`-typed field across both codebases, every filter, every DTO) — not worth it unless "admin can add a type from the UI" is an actual requirement, which hasn't been stated. Recommend confirming this before choosing B over A.

### Backend / Frontend impact (Option A, recommended)

- One migration widening the `CHECK` constraint.
- `PropertyType` union and `PROPERTY_TYPES`/`PROPERTY_TYPE_LABELS`-style arrays in `SearchPage.tsx`, `AddPropertyPage.tsx`, and anywhere else the union is exhaustively matched (`grep -rn "PropertyType" frontend/src backend/src` before building, to catch every switch/array that needs the new values — several exhaustiveness-checked switches likely exist given this codebase's TypeScript discipline).
- No DTO shape change — `propertyType` is still a string field, just a wider set of valid values.

## 3. Coupons

### Current state

No coupon/discount/promo code concept exists anywhere in the backend (confirmed — no matching code found). Purchases go through `payment_orders` (`purpose`: `listing_boost` or `premium_plan`) at a fixed `amount` sourced from `FEATURED_LISTING_PRICE_AMOUNT`/`BOOST_LISTING_PRICE_AMOUNT` env vars, no discount path at all.

### Proposed data model (additive)

```sql
CREATE TABLE coupons (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code              TEXT NOT NULL UNIQUE,      -- "LAUNCH20", stored uppercase
  discount_type     TEXT NOT NULL CHECK (discount_type IN ('percent', 'flat')),
  discount_value    INTEGER NOT NULL CHECK (discount_value > 0),  -- percent (1-100) or smallest-currency-unit flat amount
  applies_to        TEXT NOT NULL CHECK (applies_to IN ('listing_boost', 'premium_plan', 'all')),
  max_redemptions   INTEGER NULL,               -- NULL = unlimited
  redemption_count  INTEGER NOT NULL DEFAULT 0,
  valid_from        TIMESTAMPTZ NOT NULL DEFAULT now(),
  valid_until       TIMESTAMPTZ NULL,           -- NULL = no expiry
  is_active         BOOLEAN NOT NULL DEFAULT true,
  created_by        UUID NOT NULL REFERENCES users(id),  -- admin who created it
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE coupon_redemptions (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  coupon_id   UUID NOT NULL REFERENCES coupons(id),
  user_id     UUID NOT NULL REFERENCES users(id),
  payment_order_id UUID NOT NULL REFERENCES payment_orders(id),
  discount_amount_applied INTEGER NOT NULL,     -- actual amount deducted, smallest currency unit
  redeemed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (coupon_id, user_id)  -- one redemption per user per coupon; drop this constraint if multi-use-per-user is actually wanted
);

ALTER TABLE payment_orders
  ADD COLUMN coupon_id UUID NULL REFERENCES coupons(id),
  ADD COLUMN original_amount INTEGER NULL;  -- pre-discount amount, only set when a coupon was applied
```

`payment_orders.amount` keeps meaning exactly what it means today (the amount actually charged) — `original_amount` is only populated alongside a `coupon_id` so every existing reader of `.amount` (payment history, invoices, admin revenue views) needs zero changes.

### Backend

- `POST /payments/coupons/validate` — takes a code + purpose, returns the discount if valid (checks `is_active`, `valid_from`/`valid_until`, `max_redemptions`, and whether this user already redeemed it), without creating anything yet.
- `CreatePaymentOrder` use case (wherever boost/premium-plan checkout currently builds a `payment_order`) accepts an optional `couponCode`, re-validates it server-side (never trust a client-computed discount), computes the discounted `amount`, and writes `original_amount`/`coupon_id`. Insert into `coupon_redemptions` and increment `redemption_count` in the same transaction as order creation, not on payment confirmation — this needs a decision on whether a redemption should be released back if the order is later cancelled/fails (recommend: yes, decrement on `payment_orders.status` transitioning to `failed`/`cancelled`, otherwise a failed payment permanently burns a one-per-user coupon).
- Admin CRUD for coupons: `POST/GET/PATCH /admin/coupons` — straightforward, matches the existing admin CRUD patterns already in this codebase (e.g. category/moderation admin routes).

### Frontend

- Boost/premium-plan checkout flow gets a "Have a coupon?" input that calls the validate endpoint and shows the discounted total before the user pays — never silently apply an unvalidated discount.
- New admin page (`admin/CouponsPage.tsx`) for creating/editing/deactivating coupons, plumbed the same way as existing admin list pages (`admin/ReportsPage.tsx` is a reasonable template to follow).

## 4. GST

### Current state

`payments`/`invoices`/`payment_orders` have no tax fields at all — `amount` is charged and invoiced with no tax breakdown. `PAYMENT_CURRENCY=INR` is already set, so this is squarely relevant, not speculative.

### Proposed data model (additive)

```sql
ALTER TABLE payment_orders
  ADD COLUMN gst_rate    NUMERIC(5,2) NULL,   -- e.g. 18.00
  ADD COLUMN gst_amount  INTEGER NULL,        -- smallest currency unit, included within .amount
  ADD COLUMN buyer_gstin TEXT NULL;           -- optional, for B2B invoices

ALTER TABLE invoices
  ADD COLUMN gst_rate     NUMERIC(5,2) NULL,
  ADD COLUMN gst_amount   INTEGER NULL,
  ADD COLUMN taxable_amount INTEGER NULL,     -- amount - gst_amount
  ADD COLUMN buyer_gstin  TEXT NULL,
  ADD COLUMN seller_gstin TEXT NULL;          -- your business's GSTIN, likely a config value not per-row, but stored per-invoice so a historical invoice reflects the GSTIN at time of issue even if it changes later
```

All nullable, additive. Existing invoices (issued before this ships) simply have `NULL` in these columns — the invoice display/PDF (if one exists — confirm whether invoice PDFs are generated anywhere, or if `PaymentHistoryPage` only shows a list) needs to handle a `NULL` GST breakdown gracefully for pre-GST-feature invoices, showing them exactly as they display today.

### Backend

- A single source of truth for the current GST rate (config/env var, e.g. `GST_RATE_PERCENT=18`, not hardcoded, since India's GST rates for services have changed before and will again) read when an invoice is generated, not derived after the fact.
- Wherever an `invoice` row is currently created (find the exact use case — likely alongside payment confirmation in the webhook handler), compute `taxable_amount`/`gst_amount` from `amount` and the configured rate, and store all three.
- `line_description` on invoices likely needs to show the tax breakdown in the text a real accountant/user would expect (e.g. "Featured listing (7 days) — ₹169 + GST ₹31 = ₹200") — exact format needs a decision informed by what a real GST invoice legally requires (this is genuinely a compliance question, not just a UI one; **recommend involving someone with Indian GST invoicing compliance knowledge before shipping this**, since an incorrect GST invoice has real legal/tax implications beyond what code review alone can catch).

### Frontend

- `PaymentHistoryPage` / invoice display: show the tax breakdown when present, unchanged display for older invoices where the new columns are `NULL`.
- Checkout flow: show "+ GST (18%)" as a separate line before payment, not folded silently into one number.

## 5. Revenue Dashboard

### Current state

`AdminAnalyticsRepository.ts` has no revenue metric at all today — only user/property counts, view/favorite totals, and a growth chart keyed to `GROWTH_SOURCES` (users/properties over time, by the look of it). `payments` (succeeded/failed/refunded amounts) and `invoices` exist and are unused by any admin-facing aggregate.

### Proposed backend addition

A new `getRevenue()` method alongside the existing `getGrowth()` on `AdminAnalyticsRepository`, following the **exact date-bounding discipline RC1 just had to fix** (align the WHERE clause's lower bound with `generate_series`'s, both `::date`-floored — this is a lesson already paid for once in RC1's regression fixes and directly applicable here):

- Revenue by day/week/month, filtered to `payments.status = 'succeeded'` minus `'refunded'`/`'partially_refunded'` amounts, joined through `payment_orders.purpose` to break down "listing boosts" vs "premium plans" as two series, not one blended number.
- Net revenue after coupon discounts (once section 3 exists) and after GST (once section 4 exists) — i.e. this item is naturally sequenced *after* Coupons and GST in build order, or it launches without those breakdowns and gets a follow-up once they exist. Don't build a revenue dashboard that silently omits discounts/tax if those features already shipped by the time this is built.
- New endpoint: `GET /admin/analytics/revenue?days=30&groupBy=purpose` — additive, doesn't change `getGrowth()`'s existing endpoint/response shape at all.

### Frontend

- New chart on `admin/AnalyticsPage.tsx`, alongside the existing growth chart — same component/chart library already in use there (confirm which charting library `AnalyticsPage.tsx` already imports before pulling in a second one for no reason).
- Headline stat cards: total revenue (period), refund total (period), net revenue — using the same `StatCard` component the rest of the app already uses (`components/ui/StatCard.tsx`, built in Phase 4), not a new card component.

## 6. Owner Analytics — extension only

**This does not replace `MyPropertiesPage.tsx`'s existing StatCard section — it extends it.** Read `frontend/src/pages/MyPropertiesPage.tsx` before building this; the current implementation (confirmed by reading the file) computes "Total listings / Views / Favorites / Top performer" **client-side, from only the current paginated page of `propertiesApi.mine()`** — the code's own comment says totals "reflect this page only when there's more than one [page]." That's the real, specific gap to close, not a rebuild.

### What to add

- A real backend aggregate — e.g. `GET /properties/mine/stats` — returning true portfolio-wide totals (all of the owner's properties, not just the current page): total views, total favorites, total active listings, top performer by view count. This replaces the client-side `.reduce()` over one page with a real number, without changing anything else about the page's layout or the existing four `StatCard`s' positions/labels.
- A views-over-time trend, using the `property_views` table (confirmed to already exist, migration `1700000000020`, with a real per-view `viewed_at` timestamp) grouped by day for the signed-in owner's property IDs — a small line chart under the existing stat cards. This is new backend aggregation logic (a query, following the same date-bounding discipline as section 5), not a new table.
- Do **not** add a second "analytics" page or duplicate the StatCard row elsewhere — everything above is additions to the one existing section on `MyPropertiesPage.tsx`.

## 7. Device Sessions — future roadmap note only (not built in v1.1)

Per the explicit decision: **do not touch or replace** the existing "Sign out of all devices" flow (`ProfilePage.tsx`'s Security card, `logoutAllDevices()` in `AuthContext`, `POST /auth/logout-all`). This section exists only to record one finding from investigating this area, so future work doesn't have to re-discover it:

**The data model for a full per-session list already exists and needs no schema change.** The `sessions` table (migration `1700000000007`) already stores `device_id` (FK to `user_devices`), `ip_address`, `user_agent`, `created_at`, `last_active_at`, `expires_at`, `revoked_at`, `revoked_reason` per session — everything a "your active sessions: Chrome on Windows, Mumbai, last active 2 hours ago [Revoke]" UI would need is already captured on every login. The only real gap is that no endpoint currently *lists* individual sessions (only the bulk `logout-all`) and no UI renders one. When this is prioritized, it's a `GET /users/me/sessions` (list) + `DELETE /users/me/sessions/:id` (revoke one) addition plus a UI list under the existing Security card — genuinely small, deliberately left out of v1.1 per this scoping decision.
